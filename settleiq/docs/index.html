Why SettleIQ

US payments operations teams spend a large share of their day reconciling settlement across siloed tools: a core banking screen for payout status, a Snowflake query for trend analysis, an Ops dashboard for reserve balances, a separate console for ACH/Fedwire/RTP downtime, and a helpdesk queue for chargebacks. SettleIQ unifies that workflow into a chat-driven, MCP-style orchestration layer — with audit logging, sanity validation, and risk gating built in.

Deterministic in mock mode No external API calls SQLite-backed Streamlit dashboard Sanity-validated outputs Prompt-injection guard

Six-Agent Pipeline

Every user query flows through a strict, observable pipeline. Each agent has a single responsibility — classification, routing, retrieval, validation, formatting, and audit logging. Failures at any step are logged with hashes and cost estimates.

1
Classifier
Regex pre-LLM gate · intent + injection detection
2
MCP Router
Extracts date, MID, phone, domain, payout ID
3
Analyst
Lazy-loads the right skill, runs SQL on SQLite
4
Sanity Validator
$10M cap · negative-reserve check · 10s SLA
5
Formatter
Markdown · USD · ET timezone · contextual CTAs
6
Auditor
pipeline_logs · trace hash · cost estimate

Twelve Use Cases

Each use case is a complete end-to-end path through the pipeline, backed by a dedicated skill module and a deterministic test case. UC12 is the adversarial path: prompt-injection attempts are denied before they reach the router.

UC1

Settlement Details by MID

Totals, breakdown by status, net amounts, trace numbers, recent events.

UC2

Payout Journey

All settlement events grouped under a single payout ID, cross-MID view.

UC3

Phone Number Lookup

Fuzzy phone match with disambiguation when multiple merchants share a number.

UC4

Settlement Schedule

Hardcoded ACH / Fedwire / RTP timing rules and cutoff windows.

UC5

Reserve Balance

Reserve account balance with risk-tier context and recent reserve movements.

UC6

Rail Downtime

Bank and payment-rail outage windows for ACH / Fedwire / RTP / card networks.

UC7

30-Day Trend

Rolling volume, average settlement time, failure rate, return-code breakdown.

UC8

Failure Diagnostics

R01 / R03 / BANK_TIMEOUT root-cause explanations with contextual next steps.

UC9

Provisional Credit

Confirmed vs. pending provisional credits with timestamps and SLA notes.

UC10

Freeze / On-Hold

Merchant freeze status, hold reason, and risk-team context.

UC11

Chargebacks

Lookup by MID or transaction ID with reason codes (4853 / 4855 / 10.4 / …).

UC12

Injection Guard

Prompt-injection and jailbreak detection — hard denial, logged and blocked.

Synthetic US Payments Data

The repo ships with a reproducible SQLite mock dataset: 110 US merchants across verticals (rideshare, food delivery, marketplaces, hospitality), 11,000 settlement events spanning ACH / Fedwire / RTP rails, chargeback history with reason codes, and rail downtime windows. Everything is seeded deterministically in roughly one second so the system can be demoed end-to-end on a laptop with no external dependencies.

Extension Points

A USE_REAL_LLM toggle is wired in as an extension point — a real LLM provider (Anthropic, OpenAI) can be plugged into the formatter agent without touching the rest of the pipeline. The skill registry is lazy-loaded, so adding a new use case means adding a single skill module and a router pattern.

Run Locally — Three Commands

The full system runs on a laptop with no API keys and no network access. Clone the repo and run: 

# 1. Install dependencies
pip install -r requirements.txt

# 2. Seed the SQLite mock database (~1 second, deterministic)
python data/mock_generator.py

# 3. Launch the Streamlit dashboard
streamlit run dashboard/app.py

Or run a single query through the CLI: python pipeline_runner.py "Settlement details for MID01010 last 7 days". Run python pipeline_runner.py --demo to execute all twelve UCs in sequence.
